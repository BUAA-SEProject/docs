# [占位] 路线图与里程碑

> [占位] 当前文档尚未进入正式编写阶段。本页仅保留结构化草案，不可直接作为课程提交稿、开发依据或答辩材料使用。

## 里程碑总览

| 阶段 | 目标 | 主要产出 |
| --- | --- | --- |
| 计划阶段 | 明确范围、分工、风险、节奏 | 软件开发计划书 |
| 需求分析阶段 | 锁定核心需求与验收标准 | 软件需求规格说明书 |
| 系统设计阶段 | 完成架构与模块设计 | 概要设计说明书、详细设计说明书 |
| 实现与联调阶段 | 打通主链路并稳定演示 | 前后端联调版本 |
| 验收阶段 | 完成交付与答辩准备 | 测试报告、部署文档、用户手册 |

## 建议时间安排

| 时间点 | 重点工作 |
| --- | --- |
| 第 1 周 | 项目命名、范围确认、技术栈讨论、分工 |
| 第 2 周 | 完成计划书初稿、建立文档与代码仓库 |
| 第 3 周 | 完成需求分析、角色建模、核心流程设计 |
| 第 4 周 | 完成概要设计、接口规范、数据库总方案 |
| 第 5 周 | 完成详细设计、主链路编码与联调 |
| 第 6 周 | 完成测试、部署、用户手册和答辩材料 |

## 文档与代码并行策略

- 需求冻结前，不进入大规模实现。
- 详细设计完成后，再对复杂模块展开编码。
- 每次核心流程改动后，同步更新需求、设计和 API 文档。
- 每周至少进行一次文档一致性检查。

## 风险提示

- 需求范围过大，导致过程文档和实现脱节
- 自动评测和在线实验环境实现复杂度被低估
- 文档只在交付前补写，导致连续性不足
- 演示环境与开发环境不一致，影响答辩稳定性
