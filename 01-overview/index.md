# 项目概览

本部分统一回答四个问题：

1. 课程要求到底要求交付什么。
2. 本项目选择怎样的产品边界与技术路线。
3. 每份正式文档的职责是什么。
4. 项目应按什么节奏推进，才能既完成基本项又预留加分空间。

## 文档标准依据

本套文档参考以下标准与实践组织：

- ISO/IEC/IEEE 29148:2018，约束需求工程与 SRS 结构。
- ISO/IEC/IEEE 42010:2011，约束架构描述与视图组织方式。
- IEEE 1016-2009，约束软件设计说明书的设计视图与细化粒度。
- ISO/IEC/IEEE 29119 系列，约束测试计划、测试设计与测试报告组织。
- ISO/IEC 26514，约束用户文档与用户手册编写方式。

## 阅读顺序

- 先读 [课程大作业要求](course-project-requirements.md)，确认评分重点和必须提交物。
- 再读 [项目章程](project-charter.md)，确认范围、目标、推荐技术方案和开源采用策略。
- 最后读 [路线图与里程碑](roadmap-and-milestones.md)，把文档、开发、测试和答辩准备串起来。

## 文档职责地图

| 文档 | 解决的问题 | 主要输出 |
| --- | --- | --- |
| 课程大作业要求 | 教师和助教的外部要求是什么 | 评分边界、必交件、加分项 |
| 项目章程 | 本项目要做什么、不做什么、怎样做 | 项目目标、范围、成功标准、技术路线 |
| 路线图与里程碑 | 何时完成什么、何时冻结什么 | 周期计划、阶段出口、风险缓冲 |

## 使用规则

- 若范围边界发生变化，先修改 [项目章程](project-charter.md)，再同步 SRS 和设计文档。
- 若只是实现方案变化，但不影响范围边界，优先更新设计文档和 API 文档。
- 若交付节奏变化，优先更新 [路线图与里程碑](roadmap-and-milestones.md) 和团队协作文档。
