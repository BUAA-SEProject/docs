# 软件概要设计说明书

## 1. 文档信息

- 文档名称：在线教学与实训平台软件概要设计说明书
- 版本：v1.0
- 状态：设计基线
- 更新日期：2026-04-13
- 编写依据：SRS v4.0、软件开发计划书 v2.0、课程大作业要求
- 参考标准：ISO/IEC/IEEE 42010、IEEE 1016-2009

## 2. 设计目标与约束

### 2.1 设计目标

1. 支撑“平台初始化 -> 建课 -> 发任务 -> 学员提交 -> 自动评测 -> 人工批改 -> 成绩发布”的完整教学主链路。
2. 在课程项目复杂度可控前提下，提供清晰的模块边界、接口边界和部署边界。
3. 为后续数据库设计、API 设计、前后端编码和测试设计提供统一蓝图。

### 2.2 关键约束

- 系统主入口为 Web 浏览器，不建设原生客户端。
- 判题与代码执行必须隔离于业务主服务之外。
- 平台应支持独立部署和私有化部署。
- 首版优先采用模块化单体，不采用多服务分布式事务。

## 3. 系统上下文

### 3.1 外部实体

| 外部实体 | 作用 | 与本系统交互内容 |
| --- | --- | --- |
| 浏览器 | 用户操作入口 | 页面访问、表单提交、结果查看 |
| Judge0 CE / Docker 沙箱 | 执行代码与返回评测结果 | 提交代码、运行参数、评测结果 |
| 文件存储 | 存储课程资源、提交附件和导出文件 | 文件上传、下载、生命周期管理 |
| 消息通道 | 邮件、短信、企业 IM 等扩展通知 | 通知发送请求和回执 |
| 统一认证 | 可选 SSO 来源 | 用户身份信息与登录态 |

### 3.2 系统上下文图

```text
Teacher / Student / Admin
          |
       Browser
          |
     platform-web
          |
      platform-api
    /    |     |    \
Postgres Redis MinIO  Event Queue
                        |
                   judge-worker
                        |
                 Judge0 CE / Sandbox
```

## 4. 架构风格与分层

### 4.1 架构风格

- 业务层采用模块化单体，降低课程项目的运维复杂度。
- 判题执行采用异步工作进程，避免长耗时任务阻塞 Web 请求。
- 文件与通知采用异步事件驱动，增强可恢复性。

### 4.2 分层结构

| 层次 | 组件 | 职责 |
| --- | --- | --- |
| 表现层 | `platform-web` | 路由、页面、表单校验、图表展示、在线编辑 |
| API 层 | `platform-api` | 鉴权、业务编排、领域规则、接口输出 |
| 领域层 | 课程、任务、提交、评测、批改、治理模块 | 核心业务规则与状态流转 |
| 基础设施层 | PostgreSQL、Redis、MinIO、Judge0、日志与监控 | 存储、队列、执行、观测 |

| 模块 | 核心职责 | 主要用户 |

| 任务管理 | 任务说明、提交规则、评分规则、用例配置 | 教师 |

| 批改与成绩 | 人工评分、Rubric、成绩生成、发布、撤回、复核 | 教师、助教、学员 |

## 6. 核心业务流程

### 6.1 教学主链路

```text
管理员初始化平台
  -> 教师创建课程
  -> 教师发布任务
  -> 学员加入课程
  -> 学员提交代码/文件
  -> 系统异步发起评测
  -> 教师查看评测并人工批改
  -> 系统生成并发布成绩
  -> 学员查看反馈与成绩
```

### 6.2 关键时序一：提交与自动评测

```text
Student -> Web: 提交代码
Web -> API: 创建提交
API -> Postgres: 写入 submission
API -> Redis Queue: 投递 judge job
judge-worker -> Judge0: 执行评测
Judge0 -> judge-worker: 返回运行结果
judge-worker -> API/DB: 归一化结果并保存
API -> Notification: 发送评测完成事件
Student/Teacher -> API: 查询结果
```

### 6.3 关键时序二：人工批改与成绩发布

```text
Teacher -> API: 打开提交详情
Teacher -> API: 保存评语/人工分
API -> DB: 写入 review_record
API -> DB: 计算 grade
Teacher -> API: 发布成绩
API -> Notification: 发布成绩通知
Student -> API: 查看成绩与评语
```

## 7. 数据设计总览

### 7.1 核心实体

- 平台治理：`platform_configs`、`org_units`、`users`
- 课程域：`courses`、`course_members`、`course_resources`
- 任务域：`tasks`、`task_testcases`、`task_rubrics`
- 提交与评测：`submissions`、`submission_files`、`judge_runs`
- 批改与成绩：`review_records`、`grades`、`grade_rechecks`
- 消息与审计：`notifications`、`announcements`、`audit_logs`

### 7.2 数据原则

- 所有关键对象必须具备状态字段、创建时间、更新时间和操作者留痕。
- 提交、评测、成绩等关键结果采用“追加历史 + 当前有效记录”策略，不做无痕覆盖。
- 文件实体与业务实体解耦，文件仅保存对象存储引用与元数据。

## 8. 接口设计总览

### 8.1 API 风格

- 基础路径：`/api/v1`
- 数据格式：JSON，文件上传使用 `multipart/form-data`
- 认证方式：基于会话 Cookie 的 Web 鉴权，可扩展到统一认证
- 错误模型：统一错误码、消息、请求 ID

### 8.2 API 分组

| 分组 | 说明 |
| --- | --- |
| `/auth/*` | 登录、退出、当前用户、密码变更 |
| `/admin/*` | 平台配置、组织、用户、平台级概览、审计 |
| `/courses/*` | 课程、成员、资源、邀请码 |
| `/tasks/*` | 任务、测试用例、Rubric、提交规则 |
| `/submissions/*` | 提交创建、提交详情、提交历史、评测状态 |
| `/grades/*` | 批改记录、成绩发布、成绩导出、复核 |
| `/notifications/*` | 站内通知、公告、已读状态 |

## 10. 部署视图

### 10.1 最小可运行拓扑

| 节点 | 服务 |
| --- | --- |
| Web 节点 | Nginx + `platform-web` 静态资源 |
| API 节点 | `platform-api` |
| Worker 节点 | `judge-worker` |
| 数据节点 | PostgreSQL、Redis、MinIO |
| 观测节点 | Prometheus、Grafana、日志收集 |

### 10.2 部署特性

- API 与 Worker 可水平扩展。
- PostgreSQL 单实例起步，支持后续主从或托管化。
- Judge0 与 Worker 独立部署，便于按课程并发扩容。

## 11. 安全、可靠性与可观测性设计

### 11.1 安全

- 所有受保护接口必须在网关或控制器层完成认证与授权校验。
- 判题执行环境默认禁用外网访问，限制 CPU、内存、运行时间和文件系统权限。
- 审计日志记录登录、导入、任务发布、成绩发布、权限调整和导出行为。

### 11.2 可靠性

- 提交请求采用“先落库后排队”的模式，保证受理成功即不丢失。
- 判题失败不影响课程浏览与提交查询；失败任务可重试。
- 大文件通过对象存储保存，避免 API 进程本地磁盘成为单点。

### 11.3 可观测性

- 输出结构化日志，带 `requestId`、`userId`、`courseId` 等上下文。
- 核心指标包括 API 延迟、提交量、队列深度、评测失败率、成绩发布量。
- 提供教师课程概览和管理员平台概览双层仪表盘。

## 12. 设计取舍

| 设计点 | 选择 | 原因 |
| --- | --- | --- |
| 架构风格 | 模块化单体 + Worker | 复杂度可控，利于课程项目交付 |
| 判题方案 | 对接 Judge0/沙箱，而非内嵌执行 | 明确安全边界，降低实现风险 |
| 鉴权方案 | Web 会话 Cookie | 适合浏览器场景，便于角色权限管理 |
| 数据存储 | PostgreSQL 为主，Redis 为辅 | 事务建模清晰，队列与缓存分离 |
