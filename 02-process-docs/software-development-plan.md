## 2. 项目背景

根据课程大作业要求，本项目需围绕“在线教学与实训平台”选题完成过程文档与最终系统交付。课程方明确要求项目同时满足过程连续性、核心需求覆盖度、系统可用性、展示清晰度和团队分工合理性，因此本计划书不仅用于安排研发进度，也用于约束后续需求、设计、实现、测试与答辩准备的整体节奏。

在线教学与实训平台的目标，是为教师与学生提供一套覆盖课程管理、在线实验、自动评测、作业布置与批改、成绩统计等教学活动的一体化平台。项目既要体现软件工程方法，也要保证答辩时能够展示真实、稳定、连贯的教学场景。

## 3. 项目目标

### 3.1 总体目标

在课程规定周期内完成一个可演示、可部署、可答辩的在线教学与实训平台，并形成完整、连续、与实现一致的过程文档。

### 3.2 业务目标

- 为教师提供课程创建、资源管理、作业/实验发布、提交查看、批改反馈和成绩统计能力。
- 为学生提供选课、查看课程内容、完成实验、提交作业、查看评测结果和成绩反馈能力。
- 为项目团队提供可持续维护的文档、代码、评审与交付流程。

### 3.3 交付目标

- 按课程节点完成过程文档交付。
- 实现答辩主链路闭环：登录 -> 建课/选课 -> 发布作业/实验 -> 学生提交 -> 自动评测 -> 教师批改 -> 成绩统计。
- 保证系统在演示环境和小并发场景下稳定运行。
- 对加分项保留扩展空间，但不影响基本项完成。

- 成绩统计与发布
- 基础通知能力
- 部署、测试、用户手册和演示资料

### 4.2 本期优先级划分

- 用户与权限
- 课程管理

## 5. 核心需求基线

为避免后续做偏，本项目将以下能力作为核心需求基线，在需求、设计、实现、测试和答辩中均需一一对应。

### 5.1 用户与权限

- 学生/教师登录、退出
- 基于角色的页面和接口访问控制
- 用户基础资料查看与维护

### 5.2 课程管理

- 教师创建、编辑、归档课程
- 学生加入课程并查看课程内容
- 课程公告与教学资源管理

### 5.3 作业与实验管理

- 教师创建、发布、修改作业/实验
- 设置开始时间、截止时间、提交次数、评分规则
- 学生查看任务详情并完成提交

### 5.4 在线实验与自动评测

- 学生进入实验或提交入口
- 系统执行编译、运行、测试与评分
- 保存评测结果、日志和历史提交信息

### 5.5 批改与成绩统计

- 教师查看提交详情并补充评分或评语
- 系统统计课程成绩并支持导出
- 学生查看成绩和反馈

### 5.6 系统可用性

- 主链路可稳定演示
- 演示环境与交付版本一致
- 基础错误处理与日志保留可支持排障

## 7. 开发策略

### 7.1 总体策略

采用“先范围、后需求、再设计、后实现、最后验收”的推进方式，避免先写代码、后补文档导致的过程断裂问题。项目实施时坚持“主链路优先、复杂能力后置、文档与代码同步”的原则。

### 7.2 实施原则

- 优先打通最小可演示闭环，而不是一次性铺开全部功能。
- 复杂模块采用增量实现，例如先完成基础评测，再补日志与重测能力。
- 所有阶段文档应与当前代码状态对应，禁止完全脱离实现的纸面设计。
- 加分项只在 P0 能力稳定后推进。

### 7.3 版本策略

| 版本 | 目标 | 说明 |
| --- | --- | --- |
| V0.1 | 文档与工程骨架建立 | 完成仓库、GitBook、计划书、模块拆分 |
| V0.2 | 需求与概要设计完成 | 核心需求、模块边界、接口总览明确 |
| V0.5 | 主链路联通 | 登录、课程、作业、提交、评测、成绩可运行 |
| V0.8 | 演示稳定版本 | 界面、流程、数据、部署完成联调 |
| V1.0 | 最终交付版本 | 测试、部署、用户手册、答辩材料齐全 |

## 12. 质量保证计划

### 12.1 质量目标

- 核心需求覆盖完整
- 交付系统可用
- 文档与代码一致
- 演示过程稳定、可复现

### 12.2 质量控制活动

- 需求评审：检查是否覆盖课程要求和核心需求
- 设计评审：检查模块划分与接口设计是否可实现
- 代码评审：检查实现质量和风险
- 文档评审：检查内容完整性与前后连续性
- 回归测试：检查核心流程是否被改坏

### 12.3 测试策略

- 单元测试覆盖关键规则
- 接口测试覆盖高频 API
- 集成测试覆盖教师与学生主链路
- 冒烟测试覆盖答辩演示流程
