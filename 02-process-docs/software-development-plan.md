## 2. 项目背景

根据课程大作业要求，本项目围绕"在线教学与实训平台"选题，完成过程文档与最终系统交付。课程评价由过程文档和交付项目两部分构成，要求过程文档内容正确丰富、具有连续性、与最终系统对应；交付项目需满足核心需求覆盖度、系统可用性、答辩展示清晰度和团队分工合理性四项基本考核。

本平台对标头歌（educoder.net）、希冀（judge.buaa.edu.cn）等真实教学平台，为教师与学生提供课程管理、在线实验、自动评测、作业布置与批改、成绩统计等一体化能力。

## 3. 项目目标

### 3.1 总体目标

在课程规定周期内完成一个可演示、可部署、可答辩的在线教学与实训平台，并形成完整、连续、与实现一致的过程文档。

### 3.2 交付目标

- 按课程节点完成全部过程文档交付（计划书、需求、概要设计、详细设计、测试报告、部署文档、用户手册）。
- 实现答辩主链路闭环：登录 → 建课/选课 → 发布作业/实验 → 学生提交 → 自动评测 → 教师批改 → 成绩统计。
- 保证系统在演示环境和小并发场景下稳定运行，与演示视频一致。

## 4. 项目范围

### 4.1 本期范围

聚焦 Web 主端，纳入课程答辩必需能力：

- 用户登录与角色区分（教师/学生）
- 课程管理（创建、编辑、加入、查看）
- 作业与实验管理（发布、配置、提交）
- 在线实验与自动评测（编译、运行、评分）
- 批改与成绩统计（教师批改、成绩导出）
- 基础通知能力
- 部署、测试、用户手册和演示资料

### 4.2 优先级划分

#### P0 必须完成（基本项）

- 用户与权限
- 课程管理
- 作业/实验发布与提交
- 自动评测
- 批改与成绩统计
- 可演示部署版本
- 课程要求的全部过程文档

#### P1 视进度择优完成

- 通知中心完善
- 管理员基础能力
- 课程资源管理增强

#### P2 加分项候选（与课程加分项对应）

| 加分项 | 课程要求对应 |
| --- | --- |
| 抽取设计合理的 Skills | 大模型编码辅助产出的可复用 Skill |
| AI 助教 / 智能答疑 | 基于 Agent 的模块（如 AI 客服） |
| 界面统一美观 | UI/UX 美观设计 |
| 管理员仪表盘 | 交付系统的可观测性 |
| 实用拓展功能 | 有亮点且实用的拓展功能 |

### 4.3 明确不纳入

- 真实支付能力
- 大规模高并发支撑
- 完整多端同步上线

## 5. 项目组织与分工

### 5.1 分工原则

课程明确要求团队分工合理性，具体约束为：

- 平均每人至少负责 **1 个核心需求**
- 每人负责至少 **4 个模块**
- 每个模块至少 **5 个主要功能点**
- 每项核心需求有主负责人和协作人
- 每份正式文档有编写人和评审人

- 每次演示前执行文档与代码一致性检查。

## 6. 开发策略

### 6.1 总体策略

采用"先范围、后需求、再设计、后实现、最后验收"的推进方式，坚持"主链路优先、复杂能力后置、文档与代码同步"。

### 6.2 实施原则

- 优先打通最小可演示闭环，而非一次铺开全部功能。
- 复杂模块采用增量实现（如先完成基础评测，再补日志与重测）。
- 所有阶段文档应与当前代码状态对应，禁止完全脱离实现的纸面设计。
- 加分项只在 P0 能力稳定后推进。

### 6.3 版本策略

| 版本 | 目标 | 说明 |
| --- | --- | --- |
| V0.1 | 工程骨架建立 | 完成仓库、GitBook、计划书、模块拆分 |
| V0.2 | 需求与概要设计完成 | 核心需求、模块边界、接口总览明确 |
| V0.5 | 主链路联通 | 登录、课程、作业、提交、评测、成绩可运行 |
| V0.8 | 演示稳定版本 | 界面、流程、数据、部署完成联调 |
| V1.0 | 最终交付版本 | 测试、部署、用户手册、答辩材料齐全 |

## 9. 质量保证计划

### 9.1 质量目标

- 核心需求全覆盖，每条需求有页面、接口、测试证据和演示路径
- 交付系统可用，小并发下稳定无接口报错
- 文档与代码一致，过程连续可追溯
- 演示过程流畅、可复现，与演示视频一致

### 9.2 质量控制活动

| 活动 | 时机 | 关注点 |
| --- | --- | --- |
| 需求评审 | 第 8 周 | 是否覆盖课程要求和核心需求 |
| 设计评审 | 第 10/12 周 | 模块划分与接口设计是否可实现 |
| 代码评审 | 合入前 | 实现质量、安全性、与设计文档一致 |
| 文档一致性检查 | 每周 | 文档是否与当前实现对应 |
| 回归测试 | 演示前 | 核心流程是否被改坏 |

### 9.3 测试策略

- 单元测试覆盖关键业务规则
- 接口测试覆盖高频 API
- 集成测试覆盖教师与学生主链路
- 冒烟测试覆盖答辩演示流程

## 10. 配置管理计划

### 10.1 分支策略

- 主分支：`main`（稳定版本）
- 集成分支：`dev`（日常集成）
- 功能分支：`feature/*`
- 修复分支：`fix/*`

### 10.2 版本规范

- 提交格式：`type: summary`（如 `feat: 新增课程创建接口`）
- 重要节点打版本标签（V0.1 ~ V1.0）
- 演示版本和最终交付版本留存可回滚标识

### 10.3 文档管理

- 文档在 GitBook 与仓库中同步维护
- 新增页面必须同步更新 `SUMMARY.md`
- 每份正式文档保留版本号、状态、负责人、评审人

## 12. 验收与交付计划

### 12.1 过程文档验收

- 每份文档内容完整、逻辑连续、可追溯到项目实现
- 文档不得直接照搬模板或未经校验的生成内容
- 使用大模型辅助时，须单独维护使用说明并附 Session 与 Skill 材料

### 12.2 系统验收

- 核心需求均有对应页面、接口和演示路径
- 系统在演示环境稳定运行，与演示视频一致
- 小并发场景下不出现接口报错

### 12.3 答辩验收

- PPT 与演示视频清晰呈现核心需求
- 每位成员能解释自己负责的模块与文档
- 有演示失败时的兜底方案（本地环境 / 录屏）
