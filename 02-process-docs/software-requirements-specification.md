### 1.3 需求质量标准

为保证本文档可作为设计、开发、测试和评审的统一基线，需求条目应满足以下质量要求：

- 正确：准确反映业务目标和用户期望，不引入错误约束。
- 完整：覆盖触发条件、处理逻辑、异常分支、边界条件和结果状态。
- 清晰：使用统一术语，避免模糊词和多义表达。
- 一致：不同章节之间不存在相互冲突的状态、规则、口径和范围。
- 可验证：每条 MUST 级需求应能对应验收标准、验证方法或测试证据。
- 可追踪：需求应能追踪到用户场景、页面入口、数据对象和验证活动。
- 控制设计外溢：除接口边界、兼容性和约束外，不在 SRS 中展开实现方案或详细 API 设计。

## 2. 引言

### 2.1 编写目的

本说明书用于定义一体化在线教学与实训平台的业务目标、用户角色、功能范围、非功能要求、数据要求和外部接口要求，作为后续概要设计、详细设计、测试设计、上线评审和交付的统一依据。本文档重点回答以下问题：

- 平台为哪些客户和用户提供服务。
- 平台必须具备哪些可运营、可交付的核心能力。
- 平台在安全、可用性、性能、扩展性、运维与合规方面需要达到什么标准。
- 每项功能需求的验收标准是什么。

### 2.2 产品背景

在线教学、实践教学、职业培训和企业内训正在持续向平台化、云化和数据化演进。传统做法通常把课程管理、资源发布、在线实验、自动评测、人工批改和成绩分析拆散在多个系统中，带来以下直接问题：

- 教学组织依赖人工操作，课程开设、任务发布和成绩管理成本高。
- 学员需要自行配置本地环境，设备差异直接影响实训效果。
- 提交处理、评测结果和反馈数据无法形成统一闭环，难以支持规模化教学与精细化运营。
- 多机构、多课程和多用户并行使用时，缺乏统一的权限、审计、资源管控和服务质量保障机制。

本平台定位为面向高校、职业培训机构和企业培训部门的一体化在线教学与实训产品，通过统一的平台能力支撑课程组织、实验训练、代码评测、过程记录、成绩分析和运营管理。

### 2.3 产品范围

平台 V1 版本覆盖以下核心业务闭环：

1. 机构入驻并完成组织、用户和权限初始化。
2. 教师创建课程、组织资源并发布作业或实验任务。
3. 学员加入课程、查看任务、在线完成提交。
4. 系统对提交进行自动评测或流转到人工批改。
5. 教师、助教和管理者查看结果、处理反馈并发布成绩。
6. 机构管理员和平台运营方查看课程运行、资源占用、用户活跃度和关键审计信息。

以下内容不属于 V1 必备范围：

- 独立的商业订阅计费、开票和合同履约系统
- 线下硬件设备管控和考场专用终端管理
- 非教学业务场景下的通用项目管理或通用办公协作能力
- 与所有第三方教务系统的深度定制集成

## 3. 总体描述

### 3.1 产品定位

本平台是一个面向多机构运营的在线教学与实训平台，支持以 SaaS 方式交付，也支持面向单一客户的私有化部署。平台不是单门课程的辅助工具，而是需要长期承载多租户、多课程、多用户并发使用的正式业务系统。

平台核心价值包括：

- 为机构提供统一的课程、任务、实训和成绩管理入口。
- 为教师提供标准化的任务发布、自动评测和批改工作流。
- 为学员提供即开即用的在线实验与反馈体验。
- 为机构管理员和平台运营方提供可追踪、可审计、可扩展的运营基础设施。

### 3.2 目标客户与用户角色

| 角色 | 目标 | 典型操作 | 优先级 |
| --- | --- | --- | --- |
| 平台超级管理员 | 管理租户、平台级配置、资源策略和全局审计 | 创建租户、分配资源、查看全局运行状态 | MUST |
| 机构管理员 | 维护机构信息、账号、组织和课程级配置 | 导入用户、配置角色、查看机构数据 | MUST |
| 教师 | 组织课程、发布任务、查看提交、批改和分析结果 | 建课、发任务、设规则、批改、导出成绩 | MUST |
| 助教 | 在授权范围内协助批改、答疑和管理任务 | 查看提交、应用 Rubric、处理复核 | SHOULD |
| 学员 | 参与课程、完成实验、查看反馈和成绩 | 加入课程、在线编辑、提交、查看结果 | MUST |
| 平台运维/客服 | 保障平台稳定运行，处理告警、故障和客户支持 | 查看日志、监控、工单处理、异常排查 | SHOULD |

### 3.3 部署形态与运行前提

- 平台应支持浏览器访问的 Web 端作为主要交互入口。
- 平台应支持多租户 SaaS 部署形态，并允许私有化部署时通过配置关闭租户能力中的部分平台级入口。
- 平台应具备独立的应用层、数据层、文件存储层和评测执行层。
- 对编程类实训任务，平台应提供在线编辑与隔离执行能力，不应依赖学员本地环境作为基础前提。

### 3.4 业务约束与设计假设

- 租户之间的数据、配置和资源访问必须隔离。
- 平台应优先满足机构级正式运营场景，而不是短期试运行或一次性活动场景。
- 自动评测能力需优先覆盖当前主推的任务类型（编程题、填空题），并预留后续扩展多语言、多题型和多运行环境的能力。
- 具体技术栈、消息系统、容器编排方式和第三方服务选型由设计阶段确定，但需求上必须保证可扩展、可观测和可替换。
- 单次提交文件大小上限默认 50 MB，可按租户配置调整。
- 单次评测任务执行时间上限默认 60 秒，可按任务配置调整。
- 系统默认语言为中文简体，界面文案应可配置。

## 4. 业务场景与用例详述

> 本章为每个业务场景提供详细的用例描述，包含前置条件、主流程、备选流程、异常流程和后置条件。每个用例可直接作为测试用例设计的来源。

### 4.1 机构入驻与初始化

#### UC-01 创建租户并开通服务

| 项目 | 内容 |
| --- | --- |
| 场景编号 | BS-01 |
| 参与角色 | 平台超级管理员 |
| 业务目标 | 为新机构创建独立租户空间并完成基础配置 |

**前置条件：**
- 平台超级管理员已登录管理后台。
- 待创建机构尚未在平台中注册。

**主流程：**
1. 管理员进入"租户管理"页面，点击"新建租户"。
2. 填写租户名称、机构标识、联系人信息和初始管理员账号。
3. 配置租户基础资源策略（评测并发数、存储配额、可用功能模块）。
4. 提交创建请求，系统校验机构标识唯一性。
5. 系统创建租户空间、生成初始管理员账号并发送激活通知。
6. 管理员在租户列表中确认新租户状态为"试用中"。

**备选流程：**
- 3a. 管理员选择从模板导入默认配置：系统自动填充推荐资源策略，管理员可微调后提交。
- 5a. 初始管理员账号通过邮件邀请方式创建：系统发送包含激活链接的邮件。

**异常流程：**
- 4a. 机构标识已存在：系统提示"该标识已被使用"，管理员修改后重新提交。
- 5b. 邮件发送失败：系统记录失败原因，管理员可手动设置初始密码。

**后置条件：**
- 新租户记录已创建，状态为"试用中"。
- 初始管理员账号已生成或邀请已发出。
- 审计日志记录了本次操作。

#### UC-02 机构初始化

| 项目 | 内容 |
| --- | --- |
| 场景编号 | BS-02 |
| 参与角色 | 机构管理员 |
| 业务目标 | 完成组织结构配置和用户导入，使机构可以正常使用平台 |

**前置条件：**
- 租户已创建，机构管理员已激活账号并登录。

**主流程：**
1. 机构管理员进入"机构设置"页面，完善机构名称、Logo 等基础信息。
2. 进入"组织结构"页面，创建院系、班级或部门层级。
3. 进入"用户管理"页面，选择导入方式（Excel 批量导入 / 手动创建 / 邀请链接）。
4. 若选择批量导入：上传符合模板格式的 Excel 文件，系统校验格式和数据合法性。
5. 系统创建用户账号，自动分配到对应组织单元。
6. 管理员为用户分配角色（教师、学员等）。
7. 管理员确认导入结果，查看成功/失败统计。

**备选流程：**
- 3a. 通过邀请链接导入：管理员生成邀请链接并分发，用户自行注册后由管理员审批。
- 4a. 手动逐个创建：管理员填写单个用户信息并提交。

**异常流程：**
- 4b. Excel 格式错误：系统提示错误行号和原因，管理员修正后重新上传。
- 4c. 存在重复账号：系统列出冲突记录，管理员选择跳过或覆盖。
- 5a. 部分用户创建失败（如手机号格式错误）：系统返回失败明细，成功记录正常入库。

**后置条件：**
- 组织结构已建立。
- 用户账号已创建并分配角色。
- 导入结果已记录，可供管理员复查。

### 4.2 教学与实训主流程

#### UC-03 创建课程并组织资源

| 项目 | 内容 |
| --- | --- |
| 场景编号 | BS-03 |
| 参与角色 | 教师 |
| 业务目标 | 创建课程并准备教学资源，使学员可以加入和学习 |

**前置条件：**
- 教师已登录，且所属机构处于正常状态。

**主流程：**
1. 教师进入"我的课程"页面，点击"创建课程"。
2. 填写课程名称、简介、所属学期、所属组织单元。
3. 系统生成唯一课程编码和默认邀请码。
4. 教师进入课程详情，上传课程资源（课件、参考文档、视频链接）。
5. 教师组织资源到章节或模块结构中。
6. 教师添加助教（可选）。
7. 教师将课程状态从"草稿"切换为"进行中"，课程对学员可见。

**备选流程：**
- 2a. 教师选择从已有课程克隆：系统复制课程结构、资源引用和任务模板到新课程。
- 6a. 教师通过用户搜索添加助教：系统将该用户添加为课程助教角色。

**异常流程：**
- 4a. 上传文件超过大小限制：系统提示"文件大小不得超过 XX MB"，拒绝上传。
- 4b. 上传文件格式不支持：系统提示支持的格式列表。
- 7a. 课程缺少必填信息：系统提示需要补充的字段，阻止发布。

**后置条件：**
- 课程已创建，状态为"进行中"（或"草稿"待发布）。
- 资源已上传并关联到课程。
- 课程邀请码可供学员使用。

#### UC-04 发布作业或实验

| 项目 | 内容 |
| --- | --- |
| 场景编号 | BS-04 |
| 参与角色 | 教师 |
| 业务目标 | 配置并发布教学任务，使学员可以查看要求并完成提交 |

**前置条件：**
- 课程已创建且状态为"进行中"。
- 教师已登录并进入目标课程。

**主流程：**
1. 教师进入课程的"任务管理"页面，点击"新建任务"。
2. 选择任务类型：编程作业 / 文档作业 / 实验报告。
3. 填写任务标题、任务描述（支持富文本和附件）。
4. 配置提交规则：开放时间、截止时间、允许提交次数、允许的文件类型和大小。
5. 配置评分方式：
   - 自动评测：上传评测用例（输入/输出对）、选择运行语言、设置时间和内存限制。
   - 人工批改：定义 Rubric 评分项（评分维度、分值、扣分规则）。
   - 混合模式：先自动评测获取基础分，再由教师人工补充评分。
6. 教师预览任务配置，确认无误后发布。
7. 系统将任务状态变更为"已发布"，通知已加入课程的学员。

**备选流程：**
- 2a. 教师从任务模板库选择模板：系统预填模板内容，教师修改后发布。
- 5a. 教师选择不配置自动评测（纯手动批改任务）：跳过评测配置步骤。
- 6a. 教师选择"保存草稿"：任务状态为"草稿"，不对学员可见。

**异常流程：**
- 5b. 评测用例文件格式错误：系统提示格式要求，拒绝保存。
- 5c. Rubric 总分与任务满分不一致：系统警告并要求调整。
- 6b. 截止时间早于开放时间或未满足最小时间间隔：系统提示"截止时间必须晚于开放时间至少 1 小时"。
- 6c. 开放时间早于当前时间：系统允许发布，任务立即生效，并在预览页提示"发布后立即对学员开放"。

**后置条件：**
- 任务已创建并关联到课程。
- 已发布任务对课程学员可见。
- 通知已生成（如配置了通知）。

#### UC-05 参与课程并完成提交

| 项目 | 内容 |
| --- | --- |
| 场景编号 | BS-05 |
| 参与角色 | 学员 |
| 业务目标 | 加入课程、查看任务要求并完成作业提交 |

**前置条件：**
- 学员已登录。
- 目标课程存在且状态为"进行中"。
- 任务已发布且在提交时间窗口内。

**主流程：**
1. 学员通过邀请码或课程搜索找到目标课程，点击"加入课程"。
2. 系统校验邀请码有效性和学员资格，确认后将学员添加为课程成员。
3. 学员在课程任务列表中查看已发布任务。
4. 学员点击任务进入详情页，查看任务描述、提交要求和截止时间。
5. 学员选择提交方式：
   - 在线代码编辑器：在浏览器内编写代码。
   - 文件上传：选择本地文件上传。
6. 学员完成内容后点击"提交"。
7. 系统校验提交内容（格式、大小、时间窗口、剩余提交次数）。
8. 校验通过后，系统生成唯一提交 ID，记录提交时间、内容引用和提交者信息。
9. 系统返回提交成功确认，显示提交 ID 和状态。

**备选流程：**
- 1a. 学员通过组织分配自动加入课程：无需手动操作。
- 5a. 学员使用在线编辑器，中途保存草稿：系统自动保存（每 30 秒）和手动保存，下次进入可恢复。
- 6a. 学员在截止前修改提交：若剩余提交次数 > 0，可重新提交，历史提交保留。

**异常流程：**
- 1b. 邀请码无效或已过期：系统提示"邀请码无效，请联系教师获取"。
- 7a. 提交超过大小限制：系统提示"文件大小超过限制（最大 XX MB）"。
- 7b. 提交次数已用尽：系统提示"已达到最大提交次数，无法继续提交"。
- 7c. 已超过截止时间：系统提示"任务已截止，无法提交"。
- 7d. 文件格式不符合要求：系统提示"请上传 XX 格式的文件"。
- 8a. 提交保存失败（存储异常）：系统提示"提交失败，请稍后重试"，未扣减提交次数。

**后置条件：**
- 提交记录已生成，包含唯一 ID、时间戳和内容引用。
- 如任务启用自动评测，评测任务已入队。
- 学员可在"我的提交"中查看记录。

#### UC-06 自动评测与结果反馈

| 项目 | 内容 |
| --- | --- |
| 场景编号 | BS-06 |
| 参与角色 | 系统（自动）、学员（查看结果）、教师（查看概览） |
| 业务目标 | 对学员提交进行自动评测并输出结构化结果 |

**前置条件：**
- 任务已配置自动评测规则和评测用例。
- 学员提交已被系统成功受理。

**主流程：**
1. 系统检测到新提交，将评测任务加入队列，状态设为"待处理"。
2. 评测调度器从队列取出任务，分配沙箱执行环境。
3. 沙箱加载学员提交的代码、编译（如适用）并运行。
4. 系统按评测用例逐一执行：对比实际输出与期望输出。
5. 生成评测结果：总分、各用例通过/失败状态、执行时间、内存使用。
6. 评测结果写入数据库并关联到提交记录，状态变更为"已完成"。
7. 系统向学员发送评测完成通知。
8. 学员查看评测结果：得分、通过用例数、错误摘要和执行日志（如配置展示）。

**备选流程：**
- 3a. 提交内容为非编译型语言（如 Python）：跳过编译步骤，直接执行。
- 4a. 教师配置了部分用例对学员隐藏：结果仅显示"通过 X/Y"，不展示隐藏用例细节。
- 8a. 教师配置了延迟展示结果：结果在指定时间后才对学员可见。

**异常流程：**
- 3b. 编译失败：记录编译错误信息，评测状态为"失败"，返回编译错误详情给学员。
- 3c. 运行超时（超过任务配置的时间限制）：终止执行，记录"超时"，返回超时提示。
- 3d. 运行超内存限制：终止执行，记录"内存超限"。
- 3e. 运行时异常（段错误、除零等）：记录异常类型和位置信息。
- 2a. 沙箱资源不足，任务持续排队：超过 5 分钟未开始处理时，系统向运维告警。
- 6a. 结果写入失败：系统重试 3 次，仍失败则标记为"系统异常"并通知运维。

**后置条件：**
- 评测结果已持久化并关联到提交记录。
- 学员可查看结果（按可见性配置）。
- 教师可在任务概览中查看评测统计。

#### UC-07 人工批改与成绩发布

| 项目 | 内容 |
| --- | --- |
| 场景编号 | BS-07 |
| 参与角色 | 教师、助教 |
| 业务目标 | 完成人工批改、生成最终成绩并发布给学员 |

**前置条件：**
- 任务已截止或教师手动关闭提交。
- 存在待批改的提交记录。

**主流程：**
1. 教师进入任务的"批改管理"页面，查看提交列表（按提交时间或学员排序）。
2. 教师选择一个待批改提交，查看提交内容详情。
3. 若任务有自动评测结果，系统同时展示评测得分和详情。
4. 教师根据 Rubric 逐项评分：
   - 选择各评分维度的等级或填写分值。
   - 填写评语和改进建议。
5. 教师提交批改结果，系统计算最终成绩（自动评测分 × 权重 + 人工批改分 × 权重）。
6. 批改结果保存并记录批改者和批改时间。
7. 教师完成全部批改后，进入"成绩管理"页面。
8. 教师预览成绩汇总（分布图、平均分、最高/最低分）。
9. 教师点击"发布成绩"，成绩状态变更为"已发布"。
10. 系统向学员发送成绩发布通知。

**备选流程：**
- 1a. 助教进行初步批改：助教完成批改标注，教师进行复核和确认。
- 4a. 纯人工批改任务（无 Rubric）：教师直接填写总分和评语。
- 9a. 教师选择"部分发布"：仅发布已批改学员的成绩，其余待后续处理。

**异常流程：**
- 4b. 教师给出的分数超出任务满分：系统提示"分数不得超过满分 XX 分"。
- 5a. 成绩计算公式配置错误：系统提示权重之和不等于 100%，要求教师调整。
- 9b. 存在未批改提交但教师尝试发布全部成绩：系统警告"尚有 N 份提交未批改"，教师确认后可选择仅发布已批改部分。

**后置条件：**
- 批改记录已保存，包含批改者、时间、各项评分和评语。
- 最终成绩已生成。
- 成绩发布后学员可查看。

### 5.1 租户与组织管理

| 编号 | 需求 | 描述 | 优先级 |
| --- | --- | --- | --- |
| FR-TEN-01 | 租户创建 | 平台应支持创建租户，并为租户分配独立的基础配置空间 | MUST |
| FR-TEN-02 | 租户配置管理 | 平台应支持配置租户名称、标识、品牌信息、可用能力和资源策略 | MUST |
| FR-TEN-03 | 组织结构管理 | 租户内应支持维护院系、班级、团队或部门等组织结构 | MUST |
| FR-TEN-04 | 用户导入与邀请 | 机构管理员应能导入用户、手动创建用户或发送邀请 | MUST |
| FR-TEN-05 | 角色分配 | 机构管理员应能为用户分配教师、助教、学员等角色 | MUST |
| FR-TEN-06 | 数据隔离 | 不同租户之间的数据查询、文件访问和资源配置必须隔离 | MUST |
| FR-TEN-07 | 资源配额管理 | 平台应支持按租户配置评测并发、存储和任务运行等配额 | SHOULD |
| FR-TEN-08 | 机构品牌定制 | 平台可支持租户级 Logo、主题和登录页品牌信息配置 | COULD |

**验收标准：**

| 编号 | 验收条件 |
| --- | --- |
| AC-TEN-01 | 创建租户后，租户列表中可查到新租户记录，状态为"试用中" |
| AC-TEN-02 | 修改租户名称后，该租户用户登录后看到的机构名称同步更新 |
| AC-TEN-03 | 创建三级组织结构（院 → 系 → 班级），用户可被分配到任意层级节点 |
| AC-TEN-04 | 通过 Excel 导入 100 名用户，成功率 100%（数据合法时）；导入含 5 条格式错误数据时，系统正确返回 5 条错误明细，其余 95 条正常导入 |
| AC-TEN-05 | 将用户角色从"学员"变更为"助教"后，该用户立即获得助教权限入口 |
| AC-TEN-06 | 租户 A 的管理员无法通过 API 或页面查询到租户 B 的任何数据 |

**业务规则：**
- BR-TEN-01：每个租户必须拥有全局唯一的机构标识（slug），创建后不可修改。
- BR-TEN-02：组织结构最多支持 5 级层级嵌套。
- BR-TEN-03：用户必须归属于至少一个组织单元。
- BR-TEN-04：批量导入模板必须包含以下必填字段：姓名、账号（学号/工号）、组织单元、初始角色。
- BR-TEN-05：停用租户时，该租户下所有用户立即无法登录，但数据保留不删除。

### 5.2 身份认证与权限控制

| 编号 | 需求 | 描述 | 优先级 |
| --- | --- | --- | --- |
| FR-IAM-01 | 登录与退出 | 平台应支持用户安全登录、保持会话状态并安全退出 | MUST |
| FR-IAM-02 | 角色识别与授权 | 系统应基于用户角色控制页面、接口和数据权限 | MUST |
| FR-IAM-03 | 未授权访问拦截 | 未登录或无权限用户不得访问受保护资源 | MUST |
| FR-IAM-04 | 账号状态管理 | 平台应支持启用、停用、锁定和解锁账号 | MUST |
| FR-IAM-05 | 密码管理 | 平台应支持密码重置、首次登录修改和失效策略 | SHOULD |
| FR-IAM-06 | 单点登录集成 | 平台应预留 OAuth2/OIDC/SAML 等统一认证接入能力 | SHOULD |
| FR-IAM-07 | 多因素认证 | 面向高安全客户或管理员账户，平台可支持 MFA | COULD |

**验收标准：**

| 编号 | 验收条件 |
| --- | --- |
| AC-IAM-01 | 使用正确账号密码登录后，进入对应角色首页；使用错误密码登录时，返回"账号或密码错误" |
| AC-IAM-02 | 学员角色登录后，导航栏不显示"课程管理"入口；直接请求教师 API 返回 403 |
| AC-IAM-03 | 未登录用户访问课程页面时，自动重定向到登录页 |
| AC-IAM-04 | 管理员停用某账号后，该用户在 30 秒内无法继续操作（会话失效） |
| AC-IAM-05 | 用户首次登录后，系统强制要求修改初始密码才可进入功能页面 |

**业务规则：**
- BR-IAM-01：密码长度不少于 8 位，须包含字母和数字。
- BR-IAM-02：连续 5 次登录失败后，账号自动锁定 30 分钟。
- BR-IAM-03：用户会话超时时间为 2 小时（可配置），超时后需重新登录。
- BR-IAM-04：一个用户在同一租户内可拥有多个角色（如某用户既是 A 课程教师也是 B 课程学员）。
- BR-IAM-05：权限校验同时发生在前端（隐藏入口）和后端（API 拦截），仅前端隐藏不作为安全保障。

## 6. 非功能需求

### 6.1 可用性

- 平台应支持用户在浏览器中完成核心操作，不将复杂本地环境配置作为基础使用门槛。
- 关键页面应具备清晰的导航和状态反馈，避免用户在课程、任务、提交和成绩之间丢失上下文。
- 对失败操作、参数错误、权限不足、上传失败和评测失败应提供可理解提示。
- 首次使用用户应能在无培训的情况下完成"加入课程 → 查看任务 → 提交作业 → 查看结果"流程。
- 关键表单（如创建任务、配置 Rubric）应有输入校验和即时反馈，避免提交后才发现错误。
- 页面加载过程中应展示加载指示器，超过 3 秒未返回结果应展示明确等待提示。

### 6.2 性能

| 场景 | 指标 | 目标值 |
| --- | --- | --- |
| 常规页面请求（列表、详情、配置） | 95th 响应时间 | ≤ 2 秒 |
| 提交受理请求 | 返回受理结果 | ≤ 2 秒 |
| 评测状态查询 | 状态变更可被前端感知 | ≤ 5 秒 |
| 文件上传（≤ 10 MB） | 上传完成时间 | ≤ 10 秒 |
| 成绩导出（≤ 500 条） | 导出文件生成时间 | ≤ 10 秒 |
| 批量用户导入（≤ 500 条） | 导入完成反馈 | ≤ 30 秒 |
| 在线编辑器键入响应 | 字符显示延迟 | ≤ 100 毫秒 |

- 大文件上传、批量导出和长耗时评测任务应采用异步处理机制，不应阻塞交互层请求。
- 页面首屏加载时间应不超过 3 秒（网络良好条件下）。

### 6.3 可扩展性

- 平台应支持应用层、评测执行层和通知处理层的水平扩容。
- 评测任务处理能力应可根据租户规模和资源策略扩展，而不改变业务接口。
- 新增任务类型、运行环境或第三方集成时，不应要求整体重构现有核心模块。
- 新增角色类型时，权限模型应能通过配置支持，不需要修改核心权限校验逻辑。

### 6.4 可靠性与容灾

- 一旦提交被系统成功受理，不得因前端断连、页面刷新或普通服务重启导致记录丢失。
- 关键业务数据应具备备份和恢复能力。
- SaaS 部署形态下，平台月度可用性目标不低于 99.9%，计划内维护除外。
- SaaS 部署形态下，默认灾备目标为 RPO 不高于 15 分钟、RTO 不高于 4 小时；私有化部署可按合同约定调整。
- 评测服务异常不应影响平台其他功能的正常使用（如课程浏览、提交受理）。
- 系统应支持优雅降级：评测服务不可用时，提交仍可受理并排队。

### 6.5 安全性

- 所有受保护页面和接口都必须经过认证和权限校验。
- 租户之间的数据访问必须严格隔离。
- 学员提交内容必须在隔离环境中执行，并限制网络、文件系统和资源使用边界。
- 敏感数据在传输中必须加密（HTTPS），敏感凭证不得以明文方式存放在前端或公开仓库中。
- 密码存储必须使用加盐哈希（如 bcrypt），禁止明文或可逆加密。
- 平台应具备基础限流、防暴力破解和异常登录检测能力。
- API 接口应防范常见安全漏洞：SQL 注入、XSS、CSRF、文件路径遍历。
- 文件上传应校验文件类型（不仅检查扩展名，还需检查文件头部魔数），防止恶意文件上传。

### 6.6 隐私与合规

- 平台应遵循最小必要原则采集个人信息和学习行为数据。
- 租户应能够按配置执行数据保留、导出和删除策略。
- 审计日志和关键操作记录应具备留存期限配置能力。
- 若平台接入 AI 能力，应对 AI 使用范围、提示输入和输出留痕提供治理能力。
- 学员个人成绩数据不应对其他学员可见（排名展示需教师明确开启）。

### 6.7 可观测性与运维

- 平台应输出结构化日志、关键性能指标和异常告警。
- 运维人员应能查看服务健康状态、任务队列状态、评测成功率和资源使用情况。
- 关键失败链路应具备可追踪的错误上下文（请求 ID、用户 ID、时间戳），便于定位问题。
- 评测服务应暴露队列深度、平均处理时间、失败率等监控指标。
- 系统应支持日志分级（DEBUG/INFO/WARN/ERROR），生产环境默认 INFO 级别。

### 6.8 可维护性

- 功能模块、配置管理、接口协议和数据模型应保持清晰边界。
- 需求、设计、接口和测试文档应保持一致，核心流程变更时需同步更新。
- 配置项与业务代码应分离，支持不同部署环境切换。
- 数据库变更应通过版本化的迁移脚本管理，禁止手动直接修改生产数据库结构。

### 6.9 兼容性

- Web 端应兼容最新两个主要版本的 Chrome、Firefox、Edge 和 Safari。
- 页面应在 1280×720 至 3840×2160 分辨率范围内正常显示。
- 对外接口应保持一致的资源组织方式和版本兼容策略，接口变更应遵循向后兼容原则。
- 平台接口应支持后续与统一认证、对象存储、消息渠道和外部学习平台集成。

### 7.3 数据关系约束

- 一个租户可包含多个组织单元、用户和课程。
- 一个组织单元可包含多个子组织单元，且只能归属于一个租户。
- 一个课程必须归属于一个租户，并应可选关联到一个组织单元。
- 一个课程可包含多个课程成员关系和多个课程资源。
- 一个任务必须归属于一个课程。
- 一个任务可关联零个或多个 Rubric 版本，但同一时刻只能有一个生效版本。
- 一个学员可对同一任务产生多次提交，但每次提交都必须有独立记录。
- 一个提交可关联零个或一个评测结果，以及零个或多个批改记录。
- 成绩记录必须能追溯到任务、提交和批改来源。
- 一条通知记录对应一个接收用户；系统公告可扇出为多条通知记录。
- 一条审计日志对应一次已发生的关键操作事件。
- 通知和审计日志应按租户归属进行隔离和查询控制。
- 删除用户时，其历史提交、评测结果和成绩记录不得物理删除，仅逻辑标记。
- 课程与用户为多对多关系，通过课程成员关系表关联，同一用户在不同课程中可拥有不同角色。

### 7.4 状态模型要求

> 以下状态模型定义了核心对象的生命周期。状态之间的合法变更路径如下：

#### 7.4.1 租户状态

```
trial ──→ active ──→ disabled
  │                     ↑
  └─────────────────────┘
```

- trial（试用中）：新创建的租户默认状态，功能可用但可能有配额限制。
- active（有效）：正式启用状态。
- disabled（停用）：停用后所有用户无法登录，数据保留。

#### 7.4.2 课程状态

```
draft ──→ active ──→ archived
```

- draft（草稿）：创建后默认状态，仅教师可见。
- active（进行中）：学员可见，可发布任务和接收提交。
- archived（归档）：只读状态，历史数据可查询。

#### 7.4.3 任务状态

```
draft ──→ published ──→ closed
                └────────────→ closed（教师可提前手动关闭）
```

- draft（草稿）：创建后默认状态，仅教师可见。
- published（已发布）：学员可见，在时间窗口内可提交。
- closed（已关闭）：截止时间到达后自动关闭，或教师手动关闭；若由截止时间触发，前端可展示为"已截止"以便区分关闭原因。

#### 7.4.4 提交状态

```
accepted ──→ evaluating ──→ evaluated
                │
                └──→ failed
```

- accepted（待处理/已受理）：提交已被系统接收，等待进入评测或后续处理流程。
- evaluating（评测中）：评测任务正在执行。
- evaluated（已完成）：评测完成，结果可查。
- failed（失败）：评测执行异常（编译失败、系统错误等）。

#### 7.4.5 成绩状态

```
pending ──→ unpublished ──→ published ──→ withdrawn
                                │              │
                                └──────────────┘
                              （撤回后可重新发布）
```

- pending（未生成）：尚未批改或计算成绩。
- unpublished（待发布）：成绩已生成但未对学员公开。
- published（已发布）：学员可见。
- withdrawn（已撤回）：教师撤回成绩，需重新发布。

### 7.5 数据保留要求

- 提交记录、评测结果、成绩记录和审计日志应支持按租户配置保留周期。
- 在合同或监管要求范围内，平台应支持按租户导出历史数据。
- 对已删除或停用账号，应明确其历史业务记录的保留与脱敏策略。
- 默认数据保留策略：

| 数据类型 | 默认保留期限 | 过期处理 |
| --- | --- | --- |
| 提交记录与文件 | 3 年 | 文件物理删除，记录逻辑归档 |
| 评测结果 | 3 年 | 逻辑归档 |
| 成绩记录 | 5 年 | 逻辑归档 |
| 审计日志 | 1 年 | 压缩归档到冷存储 |
| 通知消息 | 90 天 | 物理删除 |
| 课程资源文件 | 跟随课程 | 课程删除时标记清理 |

### 8.2 系统对接与接口分类

| 接口对象 | 接口方向 | 作用 | 协议 | 优先级 |
| --- | --- | --- | --- | --- |
| 浏览器客户端 | 用户到平台 | 页面访问、表单提交、状态查询 | HTTPS + RESTful JSON API | MUST |
| 评测执行环境 | 平台到执行环境 | 提交任务、获取状态和评测结果 | 内部 RPC 或消息队列 | MUST |
| 文件存储服务 | 平台到存储 | 保存课程附件、提交文件和结果文件 | S3 兼容 API / 本地文件系统 | MUST |
| 数据库服务 | 平台到数据库 | 业务数据持久化 | SQL（MySQL/PostgreSQL） | MUST |
| 身份认证源 | 用户到平台或平台到认证源 | 统一认证或单点登录 | OAuth2/OIDC | SHOULD |
| 消息服务 | 平台到邮件、短信或 IM 服务 | 通知发送 | SMTP / HTTP API | SHOULD |

| 开放 API / Webhook | 平台到外部业务系统 | 数据同步、事件回调、扩展集成 | HTTPS Webhook | COULD |

#### 8.2.1 软件接口要求

- 外部软件接口应明确调用方、被调用方、输入输出数据范围、鉴权方式和失败反馈原则。
- 内部软件接口应保证模块边界清晰，避免通过数据库直连或隐式共享状态耦合核心流程。
- 与统一认证、消息服务、对象存储和第三方学习平台的接口应具备可替换性。

#### 8.2.2 通信接口要求

- 所有跨网络通信应定义超时、重试、幂等和错误码处理原则。
- 与异步处理相关的通信链路应明确消息投递、消费确认和失败补偿策略。
- 对外通信应满足传输加密、日志留痕和访问控制要求。

#### 8.2.3 硬件与执行环境接口要求

- 评测执行环境应提供标准化的任务投递、状态回传和结果回收边界。
- 平台与文件存储、数据库、消息基础设施之间的交互不应绑定特定厂商实现。
- 若后续引入本地终端、实验设备或专用硬件，应通过独立接口规范扩展，不影响现有教学主链路。

#### 8.2.4 服务接口要求

- 服务接口需支持容量扩展、访问审计和故障隔离。
- 对外开放能力应支持租户级授权和访问范围控制。
- 重要服务接口应提供可观测指标，至少包含调用量、成功率、失败率和平均响应时间。

### 8.3 核心接口能力需求

## 9. 验证与验收要求

### 9.1 验证方法说明

参考通用软件需求规格说明书实践，本文档中的需求验证采用以下方法，可单独使用，也可组合使用：

| 验证方法 | 说明 | 适用场景 |
| --- | --- | --- |
| 演示（Demonstration） | 通过角色操作、界面交互或业务演示验证需求满足情况 | 核心业务流程、角色权限、界面交互 |

## 10. 业务规则汇总

> 本章汇总全文中散布的业务规则，为研发和测试提供统一参照。规则编号格式：BR-{模块}-{序号}。

### 10.1 租户与组织规则

| 规则编号 | 规则描述 |
| --- | --- |
| BR-TEN-01 | 每个租户必须拥有全局唯一的机构标识（slug），创建后不可修改 |
| BR-TEN-02 | 组织结构最多支持 5 级层级嵌套 |
| BR-TEN-03 | 用户必须归属于至少一个组织单元 |
| BR-TEN-04 | 批量导入模板必填字段：姓名、账号（学号/工号）、组织单元、初始角色 |
| BR-TEN-05 | 停用租户时所有用户立即无法登录，数据保留不删除 |

### 10.2 认证与权限规则

| 规则编号 | 规则描述 |
| --- | --- |
| BR-IAM-01 | 密码长度不少于 8 位，须包含字母和数字 |
| BR-IAM-02 | 连续 5 次登录失败后账号自动锁定 30 分钟 |
| BR-IAM-03 | 用户会话超时时间为 2 小时（可配置），超时后需重新登录 |
| BR-IAM-04 | 一个用户在同一租户内可拥有多个角色 |
| BR-IAM-05 | 权限校验同时发生在前端和后端，仅前端隐藏不作为安全保障 |

### 10.3 课程规则

| 规则编号 | 规则描述 |
| --- | --- |
| BR-CRS-01 | 一门课程必须有且仅有一名负责教师，可以有 0 到多名助教 |
| BR-CRS-02 | 邀请码默认有效期 30 天，教师可手动刷新或关闭 |
| BR-CRS-03 | 课程状态变更路径：草稿 → 进行中 → 归档，不可逆 |
| BR-CRS-04 | 单门课程资源总大小上限 2 GB（可按租户调整） |
| BR-CRS-05 | 归档课程不可接收新提交，但可查看和导出历史数据 |

### 10.4 任务规则

| 规则编号 | 规则描述 |
| --- | --- |
| BR-TSK-01 | 任务满分默认 100 分，教师可自定义 |
| BR-TSK-02 | 截止时间必须晚于开放时间至少 1 小时 |
| BR-TSK-03 | 已发布任务修改描述时自动生成变更版本记录 |
| BR-TSK-04 | 混合评分模式下自动评测权重和人工批改权重之和必须等于 100% |
| BR-TSK-05 | 任务持久化状态路径：草稿 → 已发布 → 已关闭；截止时间到达后系统自动关闭，前端可展示为"已截止" |
| BR-TSK-06 | 默认允许提交次数不限，教师可配置上限（1~100 次） |

### 10.5 提交规则

| 规则编号 | 规则描述 |
| --- | --- |
| BR-SUB-01 | 每次提交生成全局唯一提交标识（含时间戳和序列号） |
| BR-SUB-02 | 提交受理后记录不可删除，即使后续评测失败 |
| BR-SUB-03 | 在线编辑器每 30 秒自动保存一次草稿 |
| BR-SUB-04 | 学员只能查看自己的提交；教师和助教可查看本课程全部提交 |
| BR-SUB-05 | 允许的文件类型由任务配置决定，默认：.c, .cpp, .java, .py, .js, .zip, .pdf, .docx, .md |
| BR-SUB-06 | 提交受理后不可撤销，只能通过新提交覆盖 |

### 10.6 评测规则

| 规则编号 | 规则描述 |
| --- | --- |
| BR-JDG-01 | 评测任务按提交时间 FIFO 顺序执行，同一学员多次提交不并行 |
| BR-JDG-02 | 单个评测任务默认时间限制 60 秒、内存限制 256 MB |
| BR-JDG-03 | 沙箱环境禁止网络访问、限制文件写入范围和进程创建数量 |
| BR-JDG-04 | 评测结果不可修改，只能通过重新评测生成新记录 |
| BR-JDG-05 | 评测队列积压超过 100 个任务时向运维告警 |
| BR-JDG-06 | 默认支持评测语言：C (gcc)、C++ (g++)、Java (JDK 17)、Python (3.10+) |

### 10.7 成绩规则

| 规则编号 | 规则描述 |
| --- | --- |
| BR-REV-01 | 最终成绩以最后一次有效提交为基准，教师可手动选择其他提交 |
| BR-REV-02 | 成绩状态变更路径：未生成 → 待发布 → 已发布 → 已撤回 |
| BR-REV-03 | 成绩发布后修改需记录变更原因和操作者 |
| BR-REV-04 | 成绩导出格式为 .xlsx |
| BR-REV-05 | 复核申请窗口为成绩发布后 7 天内（可配置） |
| BR-REV-06 | 助教批改的结果标记"助教批改"，教师可复核 |

### 10.8 通知规则

| 规则编号 | 规则描述 |
