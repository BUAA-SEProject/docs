## 2. 引言

### 2.1 编写目的

本说明书用于定义一体化在线教学与实训平台在正式商用场景下的业务目标、用户角色、功能范围、非功能要求、数据要求和外部接口要求，作为后续概要设计、详细设计、测试设计、上线评审和客户交付的统一依据。本文档重点回答以下问题：

- 平台为哪些客户和用户提供服务。
- 平台必须具备哪些可售卖、可运营、可交付的核心能力。
- 平台在安全、可用性、性能、扩展性、运维与合规方面需要达到什么标准。

### 2.2 产品背景

在线教学、实践教学、职业培训和企业内训正在持续向平台化、云化和数据化演进。传统做法通常把课程管理、资源发布、在线实验、自动评测、人工批改和成绩分析拆散在多个系统中，带来以下直接问题：

- 教学组织依赖人工操作，课程开设、任务发布和成绩管理成本高。
- 学员需要自行配置本地环境，设备差异直接影响实训效果。
- 提交处理、评测结果和反馈数据无法形成统一闭环，难以支持规模化教学与精细化运营。
- 多机构、多课程和多用户并行使用时，缺乏统一的权限、审计、资源管控和服务质量保障机制。

本平台定位为面向高校、职业培训机构和企业培训部门的一体化在线教学与实训产品，通过统一的平台能力支撑课程组织、实验训练、代码评测、过程记录、成绩分析和运营管理。

### 2.3 产品范围

平台 V1 商业版本覆盖以下核心业务闭环：

1. 机构入驻并完成组织、用户和权限初始化。
2. 教师创建课程、组织资源并发布作业或实验任务。
3. 学员加入课程、查看任务、在线完成提交。
4. 系统对提交进行自动评测或流转到人工批改。
5. 教师、助教和管理者查看结果、处理反馈并发布成绩。
6. 机构管理员和平台运营方查看课程运行、资源占用、用户活跃度和关键审计信息。

以下内容不属于 V1 必备范围：

- 独立的商业订阅计费、开票和合同履约系统
- 线下硬件设备管控和考场专用终端管理
- 非教学业务场景下的通用项目管理或通用办公协作能力
- 与所有第三方教务系统的深度定制集成

## 3. 总体描述

### 3.1 产品定位

本平台是一个面向多机构运营的在线教学与实训平台，支持以 SaaS 方式交付，也支持面向单一客户的私有化部署。平台不是单门课程的辅助工具，而是需要长期承载多租户、多课程、多用户并发使用的正式业务系统。

平台核心价值包括：

- 为机构提供统一的课程、任务、实训和成绩管理入口。
- 为教师提供标准化的任务发布、自动评测和批改工作流。
- 为学员提供即开即用的在线实验与反馈体验。
- 为机构管理员和平台运营方提供可追踪、可审计、可扩展的运营基础设施。

### 3.2 目标客户与用户角色

| 角色 | 目标 | 典型操作 | 优先级 |
| --- | --- | --- | --- |
| 平台超级管理员 | 管理租户、平台级配置、资源策略和全局审计 | 创建租户、分配资源、查看全局运行状态 | MUST |
| 机构管理员 | 维护机构信息、账号、组织和课程级配置 | 导入用户、配置角色、查看机构数据 | MUST |
| 教师 | 组织课程、发布任务、查看提交、批改和分析结果 | 建课、发任务、设规则、批改、导出成绩 | MUST |
| 助教 | 在授权范围内协助批改、答疑和管理任务 | 查看提交、应用 Rubric、处理复核 | SHOULD |
| 学员 | 参与课程、完成实验、查看反馈和成绩 | 加入课程、在线编辑、提交、查看结果 | MUST |
| 平台运维/客服 | 保障平台稳定运行，处理告警、故障和客户支持 | 查看日志、监控、工单处理、异常排查 | SHOULD |

### 3.3 部署形态与运行前提

- 平台应支持浏览器访问的 Web 端作为主要交互入口。
- 平台应支持多租户 SaaS 部署形态，并允许私有化部署时通过配置关闭租户能力中的部分平台级入口。
- 平台应具备独立的应用层、数据层、文件存储层和评测执行层。
- 对编程类实训任务，平台应提供在线编辑与隔离执行能力，不应依赖学员本地环境作为基础前提。

### 3.4 业务约束与设计假设

- 租户之间的数据、配置和资源访问必须隔离。
- 平台应优先满足机构级正式运营场景，而不是短期试运行或一次性活动场景。
- 自动评测能力需优先覆盖当前商业版本主推的任务类型，并预留后续扩展多语言、多题型和多运行环境的能力。
- 具体技术栈、消息系统、容器编排方式和第三方服务选型由设计阶段确定，但需求上必须保证可扩展、可观测和可替换。

| --- | --- | --- | --- |
| FR-AI-01 | 引导式错误反馈 | 平台可对运行错误或常见问题提供解释型反馈，不直接输出完整答案 | COULD |
| FR-AI-02 | 智能问答 | 平台可支持围绕课程资源和任务说明的智能问答能力 | COULD |
| FR-AI-03 | AI 输出审计 | 若启用 AI 功能，平台应记录 AI 输出日志和调用上下文摘要，便于审计 | SHOULD |

## 6. 非功能需求

### 6.1 可用性

- 平台应支持用户在浏览器中完成核心操作，不将复杂本地环境配置作为基础使用门槛。
- 关键页面应具备清晰的导航和状态反馈，避免用户在课程、任务、提交和成绩之间丢失上下文。
- 对失败操作、参数错误、权限不足、上传失败和评测失败应提供可理解提示。

### 6.2 性能

- 在合同约定容量范围内，95% 的常规页面请求响应时间应不超过 2 秒。
- 提交请求在通过前置校验后，应在 2 秒内返回受理结果和提交标识。
- 提交状态从排队到处理中、从处理中到完成的变化，应在 5 秒内可被前端查询到。
- 大文件上传、批量导出和长耗时评测任务应采用异步处理机制，不应阻塞交互层请求。

### 6.3 可扩展性

- 平台应支持应用层、评测执行层和通知处理层的水平扩容。
- 评测任务处理能力应可根据租户规模和资源策略扩展，而不改变业务接口。
- 新增任务类型、运行环境或第三方集成时，不应要求整体重构现有核心模块。

### 6.4 可靠性与容灾

- 一旦提交被系统成功受理，不得因前端断连、页面刷新或普通服务重启导致记录丢失。
- 关键业务数据应具备备份和恢复能力。
- SaaS 部署形态下，平台月度可用性目标不低于 99.9%，计划内维护除外。
- SaaS 部署形态下，默认灾备目标为 RPO 不高于 15 分钟、RTO 不高于 4 小时；私有化部署可按合同约定调整。

### 6.5 安全性

- 所有受保护页面和接口都必须经过认证和权限校验。
- 租户之间的数据访问必须严格隔离。
- 学员提交内容必须在隔离环境中执行，并限制网络、文件系统和资源使用边界。
- 敏感数据在传输中必须加密，敏感凭证不得以明文方式存放在前端或公开仓库中。
- 平台应具备基础限流、防暴力破解和异常登录检测能力。

### 6.6 隐私与合规

- 平台应遵循最小必要原则采集个人信息和学习行为数据。
- 租户应能够按配置执行数据保留、导出和删除策略。
- 审计日志和关键操作记录应具备留存期限配置能力。
- 若平台接入 AI 能力，应对 AI 使用范围、提示输入和输出留痕提供治理能力。

### 6.7 可观测性与运维

- 平台应输出结构化日志、关键性能指标和异常告警。
- 运维人员应能查看服务健康状态、任务队列状态、评测成功率和资源使用情况。
- 关键失败链路应具备可追踪的错误上下文，便于定位问题。

### 6.8 可维护性

- 功能模块、配置管理、接口协议和数据模型应保持清晰边界。
- 需求、设计、接口和测试文档应保持一致，核心流程变更时需同步更新。
- 配置项与业务代码应分离，支持不同部署环境切换。

### 6.9 兼容性

- Web 端应兼容当前主流桌面浏览器。
- 平台接口应支持后续与统一认证、对象存储、消息渠道和外部学习平台集成。

## 7. 数据需求

### 7.1 核心数据对象

| 数据对象 | 说明 | 核心内容 |
| --- | --- | --- |
| 租户 | 使用平台的机构实体 | 租户标识、名称、状态、品牌信息、配额策略 |
| 组织单元 | 租户内部组织结构 | 部门、院系、班级或团队信息 |
| 用户 | 平台登录主体 | 账号标识、角色、状态、基础身份信息 |
| 课程 | 教学活动容器 | 课程名称、描述、负责人、状态、所属组织 |
| 课程成员关系 | 用户与课程的关联 | 用户、课程、课程角色、加入状态 |
| 课程资源 | 课程下的资料和附件 | 类型、路径、访问范围、归属课程 |
| 任务 | 作业、实验或测验定义 | 标题、规则、评分方式、发布时间、状态 |
| Rubric | 批改量规 | 评分项、分值、说明、版本 |
| 提交记录 | 学员每次提交的实体 | 提交者、任务、受理时间、状态、内容引用 |
| 评测结果 | 自动评测输出 | 所属提交、状态、得分、日志摘要、错误信息 |
| 批改记录 | 人工批改过程数据 | 批改者、评语、评分、处理时间 |
| 成绩记录 | 最终可展示结果 | 用户、任务或课程、最终成绩、可见状态 |
| 通知 | 面向用户的消息实体 | 通知类型、标题、内容、已读状态、发送范围 |
| 审计日志 | 关键操作留痕 | 操作者、操作对象、操作类型、时间、结果 |

### 7.2 数据关系约束

- 一个租户可包含多个组织单元、用户和课程。
- 一个课程必须归属于一个租户，并应可选关联到一个组织单元。
- 一个任务必须归属于一个课程。
- 一个学员可对同一任务产生多次提交，但每次提交都必须有独立记录。
- 一个提交可关联零个或一个评测结果，以及零个或多个批改动作。
- 成绩记录必须能追溯到任务、提交和批改来源。
- 通知和审计日志应按租户归属进行隔离和查询控制。

### 7.3 状态模型要求

设计阶段至少需要细化以下状态模型：

- 租户状态：试用中、有效、停用
- 课程状态：草稿、进行中、归档
- 任务状态：草稿、已发布、已截止、已关闭
- 提交状态：待处理、处理中、已完成、失败
- 成绩状态：未生成、待发布、已发布、已撤回
- 通知状态：未读、已读、已归档

### 7.4 数据保留要求

- 提交记录、评测结果、成绩记录和审计日志应支持按租户配置保留周期。
- 在合同或监管要求范围内，平台应支持按租户导出历史数据。
- 对已删除或停用账号，应明确其历史业务记录的保留与脱敏策略。

## 8. 外部接口需求

### 8.1 用户界面需求

- 平台应提供面向平台超级管理员、机构管理员、教师、助教和学员的角色化界面。
- 教师端至少应包含课程管理、任务管理、提交查看、批改、成绩分析等页面。
- 学员端至少应包含课程列表、任务详情、在线编辑或上传、结果查看、成绩查看等页面。
- 管理端至少应包含租户管理、用户管理、资源配额、审计查看和运行状态查看等页面。

### 8.2 系统对接接口

| 接口对象 | 接口方向 | 作用 | 优先级 |
| --- | --- | --- | --- |
| 浏览器客户端 | 用户到平台 | 页面访问、表单提交、状态查询 | MUST |
| 评测执行环境 | 平台到执行环境 | 提交任务、获取状态和评测结果 | MUST |
| 文件存储服务 | 平台到存储 | 保存课程附件、提交文件和结果文件 | MUST |
| 身份认证源 | 用户到平台或平台到认证源 | 统一认证或单点登录 | SHOULD |
| 消息服务 | 平台到邮件、短信或 IM 服务 | 通知发送 | SHOULD |
| 第三方学习平台 | 平台到外部系统 | 课程或成绩同步、嵌入式接入 | COULD |
| 开放 API / Webhook | 平台到外部业务系统 | 数据同步、事件回调、扩展集成 | COULD |

### 8.3 导入导出接口需求

- 平台应支持用户、课程或成绩等数据的导入导出。
- 导出结果应支持结构化格式，便于机构归档或二次处理。
- 批量导入应提供校验结果和失败原因反馈。

## 9. 上线验收标准

### 9.1 核心业务验收

| 验收项 | 最低通过标准 |
| --- | --- |
| 租户与组织管理 | 能完成租户创建、组织配置、用户导入和角色分配 |
| 身份认证与权限 | 不同角色能够访问各自功能，越权访问被拦截 |
| 课程与任务管理 | 教师能够创建课程、发布任务并控制状态 |
| 提交与评测 | 学员能够提交任务，平台能够受理、处理并返回结果 |
| 批改与成绩 | 教师或助教能够批改、发布成绩并供学员查看 |
| 通知与审计 | 平台能够生成关键通知并记录关键操作审计日志 |

### 9.2 生产就绪验收

- 关键业务数据已具备备份恢复方案。
- 关键服务、评测队列和执行节点具备监控与告警能力。
- 敏感配置、权限控制和日志留痕已满足安全要求。
- 租户数据隔离、文件访问控制和提交流程隔离已通过验证。
- 关键接口、核心流程和异常场景已具备对应测试证据。

### 9.3 可追溯性要求

为支撑后续设计、测试和交付工作，本文档中的 MUST 级需求至少应满足以下追踪关系：

- 一个需求编号对应至少一个用户场景。
- 一个需求编号对应至少一个页面或操作入口。
- 一个需求编号对应至少一组接口或处理流程。
- 一个需求编号对应至少一项测试验证。
- 一个需求编号对应至少一个上线验收检查项。
