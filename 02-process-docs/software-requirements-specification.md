## 2. 引言

### 2.1 编写目的

本说明书用于明确一体化在线教学与实训平台在需求分析阶段需要回答的问题，作为后续概要设计、详细设计、接口定义、测试设计和验收检查的直接依据。本文档重点解决以下事项：

- 明确本期系统服务对象、业务范围和边界。
- 将课程大作业要求中的“核心需求覆盖”转化为可设计、可实现、可测试的系统需求。
- 把需求分析材料中的业务洞察整理为正式需求，区分基础交付要求与增强候选要求。

### 2.2 项目背景

高校在线教学与实践课程普遍存在以下问题：教师难以高效组织课程和批改大量提交，学生需要自行搭建环境且难以获得及时反馈，课程资料、实验、评测和成绩信息分散在多个系统中，导致教学流程割裂。本项目拟建设一个面向教师与学生的 Web 化平台，把课程管理、作业与实验发布、在线提交、自动评测、教师批改和成绩统计整合到同一系统中，形成完整教学闭环。

### 2.3 产品范围

本期基础交付范围围绕以下主链路展开：

1. 教师登录系统并创建或维护课程。
2. 教师发布作业或实验任务。
3. 学生加入课程、查看任务并完成提交。
4. 系统保存提交并执行自动评测或进入人工批改流程。
5. 教师查看提交结果、补充批改并确认成绩。
6. 学生查看反馈与个人成绩，教师查看课程成绩汇总。

不纳入本期基础范围的内容包括真实支付、完整多端同步、大规模高并发承诺、复杂教务深度集成、重型云桌面集群、企业级 DevOps 平台和不影响答辩主链路的功能堆叠。

## 3. 总体描述

### 3.1 产品定位

本系统定位为一体化在线教学与实训平台，服务于高校课程中的教学组织、实践训练与结果评价。系统应优先支持课程大作业要求中的基础教学闭环，而不是追求完整替代现有大型教务系统或企业级训练平台。

### 3.2 用户角色

| 角色 | 优先级 | 目标 | 典型操作 |
| --- | --- | --- | --- |
| 教师 | P0 | 组织课程、发布任务、查看提交、批改和统计成绩 | 登录、建课、发任务、看提交、批改、发布成绩 |
| 学生 | P0 | 参与课程、完成实验或作业、查看反馈和成绩 | 登录、加课、查看任务、在线编辑或上传、提交、查看结果 |
| 助教 | P1 | 协助教师批改和管理课程任务 | 查看提交、按授权批改、辅助统计 |
| 平台管理员 | P1 | 维护平台可用性和基础运行数据 | 维护账号状态、处理异常、查看日志 |

教师和学生是本期验收的核心用户；助教和平台管理员作为可选增强角色，只在不影响核心闭环的前提下纳入正式实现。

### 3.3 运行环境与使用前提

- 客户端以 Web 浏览器为主，优先支持桌面端使用场景。
- 系统需具备服务端运行环境、数据存储能力和独立的提交处理环境。
- 学生完成基础编程类任务时，不应依赖本地复杂开发环境。
- 系统允许后续扩展统一认证、站内通知、代码查重和 AI 辅助能力，但这些能力不是本期基础交付前提。

### 3.4 约束与假设

- 本项目以课程答辩和小并发教学场景为目标，不对大规模线上考试场景做正式承诺。
- 自动评测能力应优先支持课程当前实际需要的任务类型，交付版本至少完整支持一种可演示的自动评测任务。
- 功能范围必须与 [软件开发计划书](D:/Code/SEproject/docs/02-process-docs/software-development-plan.md) 保持一致，新增基础需求需先更新计划与需求文档。

## 6. 非功能需求

### 6.1 可用性

- 系统应支持教师和学生在浏览器内完成核心操作，不将本地复杂环境配置作为基础使用前提。
- 用户应能够从课程主页直接找到任务入口、提交入口、结果入口和成绩入口。
- 系统对失败操作、参数错误、权限不足和评测失败应给出可理解提示，而不是仅返回空白或无意义错误码。

### 6.2 性能

- 在课程演示和小并发测试场景下，常规查询页面响应时间应控制在 3 秒内。
- 学生提交后，系统应在 2 秒内返回“已受理”或明确失败信息。
- 自动评测任务进入处理状态后，前端应能够在 5 秒内查询到最新状态变化。
- 对常规规模的编程任务，系统应在合理时间内返回评测结论；对于执行耗时较长的任务，系统至少应能持续显示处理中状态，而不能无响应。

### 6.3 可靠性

- 系统一旦成功受理学生提交，不应因页面刷新、前端断连或普通服务重启直接丢失该提交。
- 教师批改记录和最终成绩数据必须可持久保存，异常情况下不能出现成绩回退或记录错乱。
- 平台核心链路中的单点异常不应影响已完成数据的查询与回看。

### 6.4 安全性

- 所有需要身份的页面和接口都必须经过认证和权限校验。
- 学生提交内容应在隔离环境中处理，不能直接在平台主服务进程中执行。
- 密码、密钥、连接串等敏感信息不得以明文硬编码形式暴露在前端或仓库公共区域。
- 系统应记录关键安全相关操作，如登录、发布任务、修改成绩和重新评测。
- 文件上传应限制允许的类型、大小和存储位置，避免任意文件执行和覆盖风险。

### 6.5 可维护性

- 需求、设计、接口和测试文档应保持一致，涉及核心流程变更时必须同步更新。
- 系统应按业务模块拆分功能，便于后续扩展课程资源、查重、通知或 AI 辅助能力。
- 配置项与业务代码应分离，便于部署环境切换和问题排查。

### 6.6 兼容性与扩展性

- 客户端应优先兼容当前主流桌面浏览器。
- 需求设计应允许后续接入统一认证、通知渠道、外部存储和更复杂的评测环境，但本期不要求全部落地。

| 课程 | 教学活动容器 | 课程名称、课程描述、创建者、课程状态 |
| 课程成员关系 | 用户与课程的关联 | 用户、课程、角色、加入状态 |
| 课程资源 | 课程下的说明材料或附件 | 资源名称、类型、归属课程、访问范围 |
| 作业/实验任务 | 教学任务定义 | 标题、说明、所属课程、开始/截止时间、评分方式、发布状态 |
| 提交记录 | 学生对任务的每次提交 | 提交者、所属任务、提交时间、提交内容引用、当前状态 |
| 评测结果 | 自动评测输出 | 所属提交、状态、得分、结果摘要、日志或错误信息 |
| 批改记录 | 教师人工处理结果 | 所属提交、评语、人工分数、最终确认状态 |
| 成绩记录 | 面向任务或课程的成绩汇总 | 用户、任务或课程、最终成绩、发布时间、可见状态 |
| 审计日志 | 关键操作留痕 | 操作者、操作对象、操作类型、时间、结果 |

### 7.2 数据关系约束

- 一个课程可包含多个任务和多个课程成员。
- 一个任务必须归属于一个课程。
- 一个学生可以对同一任务产生多次提交，但每次提交都必须有独立记录。
- 一个提交可对应零个或一个自动评测结果，也可对应零个或多个批改动作。
- 最终成绩应能追溯到任务配置、提交记录和批改记录。
- 已发布成绩的可见性变化必须可追踪，避免成绩展示与教师确认状态不一致。

### 7.3 状态要求

建议在设计阶段至少细化以下状态：

- 课程状态：草稿、进行中、归档
- 任务状态：草稿、已发布、已截止
- 提交状态：待处理、处理中、已完成、失败
- 成绩状态：未生成、待发布、已发布

## 8. 外部接口需求

### 8.1 用户界面需求

- 教师端至少应提供登录页、课程列表页、课程详情页、任务管理页、提交查看页、批改页、成绩统计页。
- 学生端至少应提供登录页、课程列表页、课程详情页、任务详情页、提交页、结果查看页、成绩查看页。
- 对编程类任务，提交页面应提供基础编辑或上传能力，并展示状态和结果。

### 8.2 系统接口需求

| 接口对象 | 接口方向 | 作用 | 说明 |
| --- | --- | --- | --- |
| 浏览器客户端 | 用户到平台 | 页面访问、表单提交、结果查询 | 通过 Web 协议访问，承载教师与学生操作 |
| 评测执行环境 | 平台到评测环境 | 提交任务、获取状态和结果 | 用于自动评测，必须与平台主服务隔离 |
| 文件存储 | 平台到存储 | 保存课程附件、提交文件和结果文件 | 应支持按课程和任务关联文件 |
| 认证服务 | 用户到平台或平台到认证源 | 用户身份认证 | 可采用本地账号，也可为后续统一认证预留接口 |
| 通知渠道 | 平台到用户 | 发送站内提醒或其他通知 | 本期可只保留站内能力，其他方式作为扩展 |

### 8.3 导入导出接口需求

- 教师应能够导出课程成绩汇总结果，导出格式至少支持表格化结果。
- 课程和任务的初始化数据可通过后台导入或预置方式提供，但具体形式由设计阶段确定。

## 9. 验收标准

### 9.1 基础交付验收矩阵

| 验收项 | 最低通过标准 | 验收证据 |
| --- | --- | --- |
| 登录与鉴权 | 教师和学生都能登录并进入各自页面，无权限访问会被拦截 | 页面演示、接口验证、权限测试 |
| 课程管理 | 能完成建课、加课、查看课程内容的完整流程 | 课程页面、成员数据、流程演示 |
| 作业与实验管理 | 教师能创建并发布任务，学生能看到任务要求 | 任务页面、任务配置、流程演示 |
| 在线提交 | 学生能在任务开放期间提交内容，系统能保存提交记录 | 提交记录、状态展示、异常校验 |
| 自动评测 | 至少一种任务类型能完成自动评测并返回结果 | 评测结果页面、状态流转、日志或结果摘要 |
| 批改与成绩 | 教师能补充批改并形成最终成绩，学生能查看反馈 | 批改记录、成绩页面、汇总结果 |
| 文档一致性 | 需求描述能映射到设计、测试和演示路径 | 需求编号、测试项、演示脚本对应关系 |

### 9.2 增强需求验收原则

- P1 与 P2 需求不作为本期基础交付是否达标的前置条件。
- 任何增强需求如果进入正式实现范围，必须补充页面入口、数据记录、测试证据和演示路径。
- 若增强需求未实现，不影响 P0 验收，但不得在正式提交文档中写成已完成能力。

### 9.3 可追溯性要求

为支撑后续设计和测试工作，本说明书中的 P0 需求至少应满足以下追踪关系：

- 一个需求编号对应至少一个用户场景。
- 一个需求编号对应至少一个页面或操作入口。
- 一个需求编号对应至少一组接口或处理流程。
- 一个需求编号对应至少一项测试验证。
- 一个需求编号对应至少一段答辩演示路径。
