# 验收清单

## 1. 文档验收

| 检查项 | 判定 |
| --- | --- |
| 软件开发计划书、需求规格说明书、概要设计说明书、详细设计说明书齐全 | 待最终评审 |
| 测试报告包含功能测试、浏览器 E2E、压力测试、缺陷和残留风险 | 待最终评审 |
| 部署文档、用户手册、大模型使用说明作为支撑附录齐全 | 待最终评审 |
| 各文档能独立阅读，不要求评审人员跳转外部文档才能理解结论 | 待最终评审 |
| API、数据库、前后端、判题、实验运行时口径与实际系统一致 | 待最终评审 |

## 2. 系统验收

| 检查项 | 判定 |
| --- | --- |
| 核心需求都能在系统中演示 | 待最终评审 |
| 演示视频与实际系统一致 | 待最终评审 |
| 低并发课程演示主链路稳定 | 待最终评审 |
| 管理员、教师、助教、学员权限边界正确 | 待最终评审 |
| go-judge 自动评测真实链路可用 | 待最终评审 |
| Kubernetes Web 终端真实链路可用 | 待最终评审 |
| 审计日志和平台概览可正常查看 | 待最终评审 |

## 3. 压力与容量验收

| 检查项 | 判定 |
| --- | --- |
| 写提交、评测轮询、真实 go-judge、文件上传下载、通知轮询、Web 终端专项有通过结论 | 待最终评审 |
| 500/1000 并发读请求长尾已说明为失败项，不包装为通过 | 待最终评审 |
| SSE 100/300、30 分钟 soak、文件权限负例等未执行项已说明为阻塞项 | 待最终评审 |
| 压力测试结论与功能验收结论分开表述 | 待最终评审 |
| 若答辩只做低并发演示，已明确不能宣称全平台高并发容量通过 | 待最终评审 |

## 4. 团队验收

| 检查项 | 判定 |
| --- | --- |
| 每位成员有清晰负责模块 | 待最终评审 |
| 每位成员能说明设计与实现 | 待最终评审 |
| 分工材料与实际产出一致 | 待最终评审 |

## 5. 答辩准备

| 检查项 | 判定 |
| --- | --- |
| 演示账号、数据、环境就绪 | 待最终评审 |
| 讲解顺序明确 | 待最终评审 |
| 故障预案准备完成 | 待最终评审 |
| 至少完成 3 次全链路彩排 | 待最终评审 |
