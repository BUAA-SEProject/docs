# 产品与需求

本部分从产品视角对系统定位、目标用户、核心场景、需求优先级和模块边界进行整理，是 [软件需求规格说明书](../02-process-docs/software-requirements-specification.md) 的产品化摘要和阅读入口。

## 当前状态

| 文档 | 状态 | 说明 |
| --- | --- | --- |
| 系统概述 | 已完成 | 说明产品定位、范围边界、主链路和交付目标 |
| 用户角色与使用场景 | 已完成 | 说明角色、职责边界和核心业务场景 |
| 核心需求清单 | 已完成 | 按优先级汇总 V1 基线需求和后续增强需求 |
| 模块地图 | 已完成 | 建立模块、需求、页面、数据和后续文档之间的对应关系 |
| Web 终端实验 | 已完成 | 说明环境型实验的教师、学生和会话查看路径 |

## 建议阅读顺序

1. 阅读 [系统概述](system-overview.md)，确认系统要解决的问题、V1 范围和交付目标。
2. 阅读 [用户角色与使用场景](user-roles-and-scenarios.md)，确认角色边界和关键业务流程。
3. 阅读 [核心需求清单](core-requirements.md)，确认 P0 / P1 / P2 的优先级划分。
4. 阅读 [模块地图](module-map.md)，确认模块拆分和后续设计、接口、测试文档的承接关系。
5. 如需了解环境型实验，阅读 [Web 终端实验](terminal-labs.md)。

## 与 SRS 的关系

- 本部分不替代 SRS，而是把 SRS 中分散的需求、范围和场景收敛成便于讨论、拆解和答辩的产品文档。
- 需求编号、优先级和范围边界以 [软件需求规格说明书](../02-process-docs/software-requirements-specification.md) 为准。
- 若产品方向、交付对象或范围发生变化，应先更新 SRS，再同步更新本模块。

## 使用规则

- 本部分面向产品、研发、测试和答辩准备，不写详细实现方案。
- 详细接口、数据表结构、部署步骤和测试脚本应分别进入 API、设计、部署和测试文档。
- 任何新增需求都应标注优先级，并能追溯到目标用户、业务场景和验收方式。
